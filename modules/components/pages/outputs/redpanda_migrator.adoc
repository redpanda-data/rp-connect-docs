== Multiple migrator pairs

When a pipeline uses multiple migrator pairs, set the `label` field to the same value on the paired input and output.

== Performance tuning

For high-throughput workloads, adjust the following settings:

On this output:

- `max_in_flight`: Set to the total number of partitions being migrated. Higher values provide no benefit beyond the partition count.

On the paired xref:components:inputs/redpanda_migrator.adoc#performance-tuning[`redpanda_migrator` input]:

- `partition_buffer_bytes`: Set to `2MB` to increase the per-partition buffer.
- `max_yield_batch_bytes`: Set to `1MB` to yield larger batches.

== Synchronization details

=== Topics

- Topic names are preserved from the source by default. Use the `topic` field with interpolation to rename topics at the destination.
- Topics are created at the destination with the same partition count as the source.
- Replication factor defaults to the source value. Set `topic_replication_factor` to override it.
- A serverless-aware subset of topic configuration keys is copied.
- ACL replication is optional (`sync_topic_acls`):
  - `ALLOW WRITE` ACL entries are not copied.
  - `ALLOW ALL` ACL entries are downgraded to `ALLOW READ`.
  - Resource pattern type and host filters are preserved.

=== Schema Registry

- Synced once at startup, then periodically based on `schema_registry.interval` (default: every 5 minutes). Set `schema_registry.interval: 0s` for a one-time sync only.
- Use include and exclude regex patterns to filter which subjects are migrated.
- Use the `subject` field with interpolation to rename subjects at the destination.
- Schema versions: `latest` (most recent version only) or `all` (full history). Defaults to `all`.
- Soft-deleted subjects can be included with `schema_registry.include_deleted`.
- Schema IDs can be translated to new destination IDs or preserved as-is. See `schema_registry.translate_ids`.
- Schema normalization is optional (`schema_registry.normalize`).
- Compatibility settings are copied per-subject. Schema metadata and rules are not copied in Serverless mode.

=== Consumer groups

- Synced periodically based on `consumer_groups.interval` (default: every 1 minute).
- Use include and exclude regex patterns to filter which groups are migrated.
- By default, all groups except those in `Dead` state are migrated. Set `consumer_groups.only_empty: true` to migrate only `Empty` state groups.
- Consumer offsets are translated using timestamps. This is approximate and may be imprecise when multiple records share the same timestamp.
- Offsets only move forward and are never rewound at the destination.
- Source and destination must have matching partition counts for each migrated group.

== How it works

Each migration component synchronizes on a different schedule:

- *Topics*: Synced from source on startup and every 5 minutes by default, including empty source topics. Data flows as messages are read from the input. Configure the sync interval with `sync_topic_interval`, or set it to `0s` to create topics only when data flows.
- *Schema Registry*: Synced at startup, then periodically as configured.
- *Consumer groups*: Synced in the background, filtered to the topics being migrated.

== Guarantees

The migrator upholds the following guarantees:

- Topics are created at the destination with the intended partition count and replication factor.
- Existing destination topics are not overwritten. The migrator logs any partition count mismatches.
- Consumer group offsets are never rewound.
- ACL replication never grants write access at the destination.

== Limitations

- The destination Schema Registry must be in `READWRITE` or `IMPORT` mode.
- Offset translation is best-effort.
- Consumer group migration requires identical partition counts at source and destination.
